- **Reference implementation:** the **faceted matcher** shipped as part

  `agent-web` (`/match` form UI). Deterministic structured filter, no

  maintained by PWM at
  `pwm-team/coordination/agent-coord/interfaces/matching/acceptance_harness.jsonl`

  This is the same file the reference matcher's tests run against.

## What you build

A matcher that takes a natural-language prompt describing an
inverse-problem instance — in any language, phrased however a
scientist or technician would phrase it — and returns a ranked list
of PWM benchmarks that might solve it, with per-candidate rationales.

**Example prompt:**

> "I have 45 X-ray projections of a steel casting, ~256×256, moderate
> metal artifact. What can PWM reconstruct with? I want better than
> ~28 dB PSNR."

**Expected output:** ranked list with L3-029 (CT, moderate-tier) at
rank 1; a short rationale per candidate; optional Ω disambiguation
question.

Three subsystems:

1. **Prompt interpreter** — parses the user's prompt into PWM taxonomy
   (domain, modality, input shape, mismatch-parameter ranges, quality
   threshold). May use any LLM provider or local model.
2. **Card retrieval** — reads the canonical benchmark-card index
   (`benchmark_cards/L3-NNN.yaml` generated from L3 JSON; always fresh
   in the repo). Retrieves candidates using any mix of structured
   filter, embedding search, and LLM-in-the-loop ranking.
3. **Rationale generator** — per-candidate 1-2 sentence explanation
   of why this benchmark matches (or doesn't) the prompt. Must
   reference the card's fields, not hallucinate content.

## Interface contract

- **Card index:** `pwm-team/pwm_product/benchmark_cards/L3-*.yaml`
  (generated by the reference card generator). Format per
  `pwm_overview1.md §8.1`.
- **Addresses + ABIs:** same as other bounties —
  `interfaces/addresses.json` and `interfaces/contracts_abi/`. Your
  matcher is read-only with respect to chain state; no transactions.
- **Acceptance harness:** JSONL, one prompt per line:
  ```json
  {"prompt": "…", "expected_benchmark_ids": ["L3-003"],
   "expected_tier_hint": "T3_moderate", "domain": "imaging"}
  ```
  The harness is versioned; lock to the git SHA you built against.
- **Output schema:** your matcher returns JSON:
  ```json
  {"candidates": [{"benchmark_id": "L3-003", "rank": 1,
                   "confidence": 0.87,
                   "rationale": "..."}, ...],
   "clarifying_question": "...",
   "confidence_floor_hit": false}
  ```

No contract calls, no wallet needed — you are off-chain entirely.

## What must pass

1. **Harness accuracy.** ≥ **80%** of acceptance-harness prompts must
   have their `expected_benchmark_ids[0]` appear in your top-3
   candidates. Measured at every harness version the qualifying window
   spans. (The reference faceted matcher is expected to reach ~65% on
   the same harness — you are required to meaningfully beat it.)

2. **Rationale honesty.** Per-candidate rationales must reference
   fields that exist on the card. A spot-check of 20 random outputs
   must have ≥ 18 factually grounded rationales (agent-coord
   samples).

3. **No hallucinated benchmarks.** Outputs must only reference benchmark
   IDs that exist in `addresses.json[<network>]` or in the card index.
   Any hallucinated ID voids the award.

4. **Latency.** P50 under **3 seconds**, P95 under **10 seconds** on
   a reasonable reference workload (acceptance harness run
   sequentially). If you use an LLM API, cache aggressively.

5. **Availability.** If hosted, ≥ 99% uptime across the 30-day shadow
   run. If shipped as a library (e.g., a npm / pypi package with
   local LLM support), pass the harness in reproducible offline
   mode.

6. **Graceful no-match.** When no candidate clears a reasonable
   confidence floor, return `confidence_floor_hit: true` and a
   clarifying question. Do not force a bad match.

## What you may change

- **LLM provider:** OpenAI, Anthropic, Google, open-source local
  models (Llama, Mistral, etc.). Your choice; document it.
- **Retrieval strategy:** pure structured filter, pure embedding
  search, or any hybrid. Embeddings may come from any provider.
- **Caching layer:** Redis, SQLite, in-memory; any.
- **Interface:** HTTP API, Python SDK, CLI wrapper — at least ONE
  must be callable by the acceptance-harness runner.
- **Language:** matchers must handle English for the acceptance
  harness. Supporting additional languages is encouraged (and
  differentiating) but not required.

## What you may not change

- The canonical benchmark-card schema (read-only input).
- The acceptance-harness format (submit improvements via PR to the
  harness; agent-coord reviews quarterly).
- The output JSON schema above — third parties downstream rely on it.

## Shadow run

30 days after acceptance. agent-coord samples:

- 50 harness prompts re-run weekly — your accuracy must stay ≥ 80%.
- 10 synthetic "gap" prompts per week (i.e., prompts where the true
  answer is that no current benchmark fits). You must return
  `confidence_floor_hit: true` on ≥ 8 of 10; forced matches count
  against you.
- 5 adversarial prompts per week designed to trigger hallucination
  (e.g., "Reconstruct 4D CBCT with organ motion" when no such
  benchmark exists). Any hallucinated benchmark ID triggers a
  full review and possible clawback.

Three unexplained regressions void the award.

## Why this is a bounty and not a PWM reference impl

Per `pwm_overview1.md §8.3` and `MVP_FIRST_STRATEGY.md §10.2 / §11
U1b`: PWM maintains the faceted (LLM-free) floor as its reference
matcher so the protocol never has a recurring external-API
dependency. The LLM-routed layer is where specialization,
multilingual support, and prompt engineering live — areas with no
PWM comparative advantage. Third parties compete here; PWM maintains
the shared harness + card index as public infrastructure.

## What happens after qualifying

Qualifying matchers get listed on the PWM web explorer's "Third-party
matchers" page, in order of accuracy-at-submission (broken by
earliest qualifying date). The PWM-hosted faceted floor stays on the
main `/match` route; qualifying third-party matchers are linked
alongside with clear attribution. No exclusivity — multiple qualifying
matchers can coexist.
